1. Introducción

El presente documento especifica los requerimientos del Sistema Escolar CEICAVS, una plataforma web de gestión escolar para el Centro Escolar CEICAVS. El sistema centraliza las operaciones administrativas y pedagógicas en una sola aplicación accesible desde cualquier navegador moderno.

Este documento sirve como referencia de alcance para el equipo de desarrollo, el Product Owner y los stakeholders del proyecto. Cada requerimiento es trazable a una o más historias de usuario (ver documento Historias de Usuario).

4. Requerimientos Funcionales

RF-01 a RF-05: Autenticación y Autorización

IDDescripciónPrioridadHistoria
RF-01 El sistema debe autenticar usuarios mediante correo electrónico y contraseña, emitiendo tokens JWT de acceso (15 min) y refresco (7 días). Must US-001
RF-02 El sistema debe renovar el token de acceso automáticamente usando el token de refresco antes de que expire la sesión. Must US-001
RF-03 El sistema debe controlar el acceso a cada operación mediante la librería CASL según el rol del usuario autenticado. Must US-002
RF-04 El frontend debe ocultar rutas y elementos de interfaz no permitidos para el rol activo usando el componente <Can>. Must US-002
RF-05 El backend debe rechazar con HTTP 403 cualquier operación que el rol del usuario no tenga permitida, independientemente del frontend. Must US-002

RF-06 a RF-10: Gestión de Personas

IDDescripciónPrioridadHistoria
RF-06 El administrador debe poder crear, editar y eliminar (soft delete) cuentas de usuario con nombre, correo, contraseña cifrada y rol. Must US-006
RF-07 El sistema debe permitir la importación masiva de usuarios desde archivos CSV con validación de formato y detección de duplicados. Should US-007
RF-08 El administrador y el docente deben poder crear grupos, asignar nombre y agregar o remover miembros. Must US-008
RF-09 La lista de usuarios debe soportar búsqueda por nombre o correo y filtrado por rol. Should US-006
RF-10 El sistema debe permitir la eliminación en masa de usuarios seleccionados. Could US-006

RF-11 a RF-16: Sistema de Asistencia

IDDescripciónPrioridadHistoria
RF-11 El docente debe poder registrar asistencia para cada estudiante de un grupo con estado: Presente, Ausente, Tardanza o Justificado. Must US-009
RF-12 El sistema debe almacenar la fecha de cada registro con la hora del servidor para evitar manipulación. Must US-009
RF-13 El sistema debe permitir ver reportes de asistencia por período (semanal, mensual, personalizado) para un grupo. Must US-010
RF-14 El estudiante solo puede ver su propia fila en el reporte de asistencia del grupo. Must US-011
RF-15 El sistema debe generar un historial detallado de asistencia por estudiante individual. Should US-011
RF-16 El sistema debe permitir exportar el reporte de asistencia en formato CSV o PDF. Should US-012

RF-17 a RF-22: Blog Institucional

IDDescripciónPrioridadHistoria
RF-17 El sistema debe mostrar un feed de publicaciones publicadas con filtrado por categoría y paginación. Must US-013
RF-18 Administradores y docentes deben poder publicar artículos directamente; los estudiantes deben enviarlos para revisión. Must US-014, US-015
RF-19 El sistema debe mantener una cola de publicaciones pendientes de revisión accesible para admin y docentes. Should US-016
RF-20 El sistema debe notificar al estudiante cuando su publicación es aprobada o rechazada (visible en la sección «Mis Borradores»). Should US-016
RF-21 Los usuarios autenticados deben poder reaccionar a publicaciones con emojis y agregar comentarios con anidación de un nivel. Could US-017
RF-22 El sistema debe permitir crear, editar y eliminar categorías del blog (solo admin y docente). Should US-018

RF-23 a RF-26: Herramientas Educativas

IDDescripciónPrioridadHistoria
RF-23 El sistema debe ofrecer un catálogo con 14 herramientas educativas accesibles desde el navegador sin instalación adicional. Must US-019
RF-24 El usuario debe poder marcar herramientas como favoritas para acceso rápido desde el dashboard. Could US-019
RF-25 El Organizador de Tareas y las Notas Rápidas deben sincronizar su estado en la base de datos por usuario. Should US-020
RF-26 El Selector de Estudiante Al Azar debe leer los grupos del usuario autenticado para seleccionar de forma aleatoria. Should US-021

RF-27 a RF-30: Transcripción con Inteligencia Artificial

IDDescripciónPrioridadHistoria
RF-27 El sistema debe transcribir archivos de audio (MP3, WAV, M4A) directamente en el navegador usando el modelo Whisper ONNX via Web Worker. Must US-024
RF-28 El progreso de la transcripción debe mostrarse en tiempo real al usuario sin bloquear la interfaz. Must US-024
RF-29 El backend debe generar un resumen estructurado (resumen, puntos clave, elementos de acción) de la transcripción usando un proveedor LLM (Groq como primario, Gemini como respaldo). Must US-026
RF-30 El usuario debe poder editar el texto de la transcripción y guardar los cambios. Should US-027

5. Requerimientos No Funcionales

Rendimiento

IDDescripción
RNF-01Las páginas de la aplicación deben cargarse en menos de 3 segundos en una conexión de 10 Mbps.
RNF-02Las consultas GraphQL deben responder en menos de 500ms para operaciones de lectura simples.
RNF-03La transcripción de un audio de 1 minuto no debe superar los 90 segundos de procesamiento en el navegador.

Seguridad

IDDescripción
RNF-04Las contraseñas deben almacenarse hasheadas con bcrypt (factor de costo mínimo 10).
RNF-05Toda la comunicación entre el cliente y la API debe realizarse mediante HTTPS.
RNF-06El sistema debe aplicar validación de entrada en el backend para prevenir inyecciones y datos malformados.
RNF-07La autorización CASL debe verificarse tanto en el frontend (UX) como en el backend (seguridad), sin depender exclusivamente de uno.

Disponibilidad y Despliegue

IDDescripción
RNF-08El frontend debe estar desplegado en Vercel con disponibilidad mínima del 99% (SLA de la plataforma).
RNF-09El backend debe estar desplegado en Render con reinicio automático ante caídas.
RNF-10La base de datos PostgreSQL debe ser accesible desde el backend mediante variables de entorno seguras (no hardcoded).

Usabilidad

IDDescripción
RNF-11La interfaz debe ser completamente en español y adaptada al contexto educativo del Centro Escolar CEICAVS.
RNF-12La aplicación debe ser responsiva y usable en pantallas desde 1024px de ancho.

6. Restricciones y Supuestos

Restricciones

  • La plataforma no incluye recuperación automática de contraseñas (el administrador debe realizarlo manualmente).
  • No se implementará versión móvil nativa (iOS/Android); solo aplicación web responsiva.
  • La transcripción funciona únicamente en navegadores con soporte para WebGPU o WASM (Chrome 113+, Firefox 113+).
  • No se incluye notificaciones en tiempo real (WebSockets); el estado se actualiza por recarga o polling.
  • El idioma de la plataforma es exclusivamente español; no hay selector de idioma.

Supuestos

  • Los usuarios del sistema cuentan con dispositivos con conexión a internet y navegador moderno.
  • La administración de las API keys (Groq, Gemini) es responsabilidad del administrador técnico del Centro Escolar.
  • Los datos existentes del Centro Escolar (alumnos, grupos) se migrarán manualmente o mediante importación CSV.
  • Las credenciales de producción (DATABASE_URL, JWT secrets) se configuran en las variables de entorno de Vercel y Render.
